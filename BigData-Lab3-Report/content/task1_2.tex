\section{Task 1.2: State-level Median Variety per Month}

\subsection{Phân tích truy vấn (Query Analysis)}
Bài toán yêu cầu tính \textbf{median variety} cấp bang cho mỗi tháng. Cụ thể, với mỗi cặp \texttt{(ship-state, month)}, ta cần xác định giá trị trung vị (median) của ``variety'' trên tập các style thỏa điều kiện đã phục vụ ít nhất một size từ XXL trở lên.

\textbf{Định nghĩa variety.} Variety của một style trong phạm vi \texttt{(state, month)} là số lượng \texttt{SKU} phân biệt (distinct) gắn với style đó trong cùng phạm vi. Variety \textit{không phải} số order hay số size, mà là số mã sản phẩm duy nhất đã xuất hiện cho style đó tại bang và tháng xét.

\textbf{Điều kiện qualifying style.} Một style được xem là qualify (hợp lệ) nếu trong cùng phạm vi \texttt{(state, month, style)} tồn tại \textit{ít nhất một} record có cột \texttt{Size} thuộc nhóm từ XXL trở lên. Các size code được hỗ trợ bao gồm: \texttt{XXL}, \texttt{2XL}, \texttt{XXXL}, \texttt{3XL}, \texttt{4XL}, \texttt{5XL}, \texttt{6XL}. Ngoài ra, hàm kiểm tra trong code còn sử dụng biểu thức chính quy \verb|([2-9]|\d{2,})XL| để bao phủ các size lớn hơn nữa (ví dụ \texttt{7XL}, \texttt{10XL}).

\textbf{Điểm quan trọng.} Sau khi một style được xác nhận là qualify, variety của style đó vẫn phải đếm \textit{tất cả} các SKU distinct trong phạm vi \texttt{(state, month)}, kể cả các SKU có size nhỏ hơn XXL. Điều kiện size $\geq$ XXL chỉ dùng để quyết định style có được đưa vào tập tính median hay không, không dùng để lọc SKU.

\textbf{Cách tính median.} Sau khi thu được danh sách variety của tất cả qualifying styles trong một \texttt{(state, month)}, danh sách này được sắp xếp tăng dần. Nếu số phần tử lẻ, median là phần tử ở vị trí giữa; nếu chẵn, median là trung bình cộng của hai phần tử giữa. Kết quả được làm tròn đến đúng 4 chữ số thập phân.

\textbf{Schema output.} File kết quả \texttt{Task\_1-2.csv} có 3 cột:

\begin{center}
\begin{tabular}{lll}
\toprule
\textbf{Cột} & \textbf{Kiểu dữ liệu} & \textbf{Ý nghĩa} \\
\midrule
\texttt{ship-state} & String & Bang/tỉnh đã chuẩn hóa viết hoa \\
\texttt{month} & String & Tháng, định dạng \texttt{YYYY-MM} \\
\texttt{median\_variety} & Double (4 d.p.) & Median variety của các qualifying styles \\
\bottomrule
\end{tabular}
\end{center}

\subsection{Phân rã thành các bước cơ bản (Decomposition)}
Lời giải được triển khai theo tư duy MapReduce trên Spark RDD, gồm các bước sau:

\begin{enumerate}
    \item \textbf{Đọc input và chọn cột.} Chương trình đọc file CSV bằng \texttt{spark.read.csv} với header. Chỉ giữ lại 5 cột cần thiết: \texttt{Date}, \texttt{SKU}, \texttt{Style}, \texttt{Size}, \texttt{ship-state}. Các dòng có bất kỳ cột nào bị null đều bị loại bằng \texttt{.na.drop()}. Cột \texttt{ship-state} và \texttt{Size} được trim khoảng trắng và chuẩn hóa viết hoa. Cột \texttt{Date} được parse từ định dạng \texttt{MM-dd-yy} sang \texttt{YYYY-MM}; các dòng có ngày không hợp lệ bị bỏ qua.

    \item \textbf{Map sang khóa style.} Mỗi record hợp lệ được emit thành cặp key-value với key là \texttt{(state, month, style)} và value là \texttt{(SKU, isAtLeastXXL)}. Đây là bước Map đầu tiên, phát tán dữ liệu theo khóa composite 3 thành phần để chuẩn bị cho bước gom nhóm tiếp theo.

    \item \textbf{Reduce lần 1 -- tính variety và cờ XXL.} Sử dụng \texttt{aggregateByKey} với accumulator là một \texttt{mutable.Set[String]} (chứa các SKU distinct) và một cờ boolean \texttt{hasXXL}. Hàm \texttt{seqOp} thêm SKU vào set và OR giá trị \texttt{isAtLeastXXL} vào cờ. Hàm \texttt{combOp} hợp nhất hai set và OR hai cờ lại với nhau. Kết quả: mỗi key \texttt{(state, month, style)} cho ra \texttt{(variety = |set|, hasXXL)}.

    \item \textbf{Lọc style qualify.} Chỉ giữ lại các style có \texttt{hasXXL = true}. Bước filter này được thực hiện \textit{trước} khi đổi khóa để giảm lượng dữ liệu cần shuffle trong bước tiếp theo.

    \item \textbf{Map sang khóa state-month.} Đổi key từ \texttt{(state, month, style)} thành \texttt{(state, month)}, value là \texttt{variety} (số nguyên). Thành phần \texttt{style} trong key bị loại bỏ vì từ bước này trở đi ta chỉ cần gom variety theo bang và tháng.

    \item \textbf{Reduce lần 2 -- tính median.} Dùng \texttt{groupByKey} để gom tất cả variety của các qualifying styles cùng \texttt{(state, month)} thành một danh sách. Danh sách được sort tăng dần rồi truyền vào hàm \texttt{computeMedian} để tính giá trị trung vị, kết quả được format thành chuỗi 4 chữ số thập phân.

    \item \textbf{Xuất CSV.} Kết quả được \texttt{sortBy} theo \texttt{(state, month)} rồi \texttt{collect()} về driver. Chương trình dùng \texttt{PrintWriter} của Java I/O để ghi header và từng dòng kết quả ra một file CSV duy nhất, tránh tạo thư mục part files đặc trưng của Hadoop.
\end{enumerate}

\subsection{Lý do phân rã}
Bài toán đòi hỏi hai tầng aggregation vì hai lý do chính. Thứ nhất, cùng một style có thể chứa cả SKU size nhỏ (S, M, L) lẫn SKU size lớn (XXL, 3XL). Nếu lọc chỉ giữ các dòng có size $\geq$ XXL trước khi đếm variety, ta sẽ mất các SKU size nhỏ và sai định nghĩa variety. Do đó, reducer đầu tiên phải \textit{đồng thời} giữ toàn bộ tập SKU distinct \textit{và} ghi nhận style đã từng phục vụ size từ XXL trở lên.

Thứ hai, bài toán yêu cầu tính median -- một phép toán toàn cục trên danh sách đã sort, không thể thực hiện bằng một phép reduce đơn giản kiểu cộng dồn. Vì vậy cần tầng aggregation thứ hai: gom tất cả variety theo \texttt{(state, month)}, sort danh sách rồi trích giá trị trung vị. Cách phân rã thành hai tầng reducer phù hợp với mô hình key-value processing vì mỗi reducer chỉ cần xử lý dữ liệu cùng key mà không phụ thuộc vào các key khác.

\textbf{Xác định size threshold.}
\begin{lstlisting}[language=Scala]
def isAtLeastXXL(size: String): Boolean = {
  val s = size.trim.toUpperCase
  val xlSizes = Set("XXL", "2XL", "XXXL", "3XL",
                     "4XL", "5XL", "6XL")
  xlSizes.contains(s) || s.matches("([2-9]|\\d{2,})XL")
}
\end{lstlisting}

Hàm \texttt{isAtLeastXXL} trước tiên kiểm tra size có thuộc tập các giá trị phổ biến hay không. Nếu không, biểu thức chính quy sẽ bắt các size dạng \texttt{nXL} với $n \geq 2$ hoặc số có từ hai chữ số trở lên, đảm bảo bao phủ mọi size từ XXL trở lên mà không nhận lầm các giá trị như \texttt{XL} hay \texttt{1XL}.

\textbf{Map và Reduce lần 1 -- tính variety per style.}
\begin{lstlisting}[language=Scala]
val mapped = df.select("Date", "SKU", "Style", "Size",
                       "ship-state")
  .na.drop().rdd
  .flatMap(row => {
    val dateStr = row.getAs[String]("Date")
    val sku     = row.getAs[String]("SKU")
    val style   = row.getAs[String]("Style")
    val size    = row.getAs[String]("Size").trim.toUpperCase
    val state   = row.getAs[String]("ship-state")
                     .trim.toUpperCase
    parseMonth(dateStr).map(month =>
      ((state, month, style), (sku, isAtLeastXXL(size)))
    )
  })

val styleVariety = mapped.aggregateByKey(
    (scala.collection.mutable.Set.empty[String], false))(
  (acc, value) => {
    acc._1 += value._1
    (acc._1, acc._2 || value._2)
  },
  (acc1, acc2) => {
    acc1._1 ++= acc2._1
    (acc1._1, acc1._2 || acc2._2)
  }
).map {
  case ((state, month, _), (skuSet, hasXXL)) =>
    ((state, month), (skuSet.size, hasXXL))
}
\end{lstlisting}

\textbf{Reduce lần 2 -- lọc qualifying styles và tính median.}
\begin{lstlisting}[language=Scala]
val medianVariety = styleVariety
  .filter { case (_, (_, hasXXL)) => hasXXL }
  .map { case (k, (size, _)) => (k, size) }
  .groupByKey()
  .flatMap { case ((state, month), sizesIter) =>
    val varieties = sizesIter.toList.sorted
    computeMedian(varieties).map(median =>
      (state, month, median))
  }
\end{lstlisting}

\textbf{Hàm tính median.}
\begin{lstlisting}[language=Scala]
def computeMedian(sortedValues: List[Int]):
    Option[String] = {
  if (sortedValues.isEmpty) None
  else {
    val n = sortedValues.size
    val median =
      if (n % 2 == 1) sortedValues(n / 2).toDouble
      else (sortedValues(n / 2 - 1) +
            sortedValues(n / 2)) / 2.0
    Some(f"$median%.4f")
  }
}
\end{lstlisting}

\textbf{Xử lý dữ liệu lỗi.} Các dòng có giá trị null ở bất kỳ cột nào trong 5 cột cần thiết đều bị loại bằng \texttt{.na.drop()} ngay từ đầu pipeline. Hàm \texttt{parseMonth} trả về \texttt{None} khi chuỗi ngày không đúng định dạng \texttt{MM-dd-yy}, và \texttt{flatMap} tự động bỏ qua các \texttt{None} này. Cột \texttt{ship-state} và \texttt{Size} được \texttt{trim} và chuyển \texttt{toUpperCase} để chuẩn hóa, tránh trường hợp cùng một bang nhưng bị viết khác kiểu chữ (ví dụ ``Bihar'' và ``BIHAR'') dẫn đến trùng lặp key.
